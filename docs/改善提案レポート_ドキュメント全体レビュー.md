# 学習用ドキュメント改善提案レポート

**作成日：** 2025年2月25日  
**対象：** docs/ 配下の学習用ドキュメント群全体  
**査読基準：** document-reviewer.mdc、document-maker.mdc、document-planner.mdc

---

## 1. エグゼクティブサマリー

本レポートは、非エンジニア向けWebディレクター学習ドキュメント群を深く読み込み、表層的な表記から構成・技術的正確性・対象読者適合性まで多角的に検証した結果をまとめたものです。

**主な発見：**
- 全体として高品質で、対象読者層に沿った丁寧な説明と豊富な例え話が効果的
- 用語・数値の不整合が複数箇所で確認された
- 一部の技術記述に修正が必要
- ルール遵守状況には改善余地あり

## 2. 用語・表記の一貫性

### 2.1 「修得」と「習得」の混在

**現状：**
- 「あなた（修得前）」「あなた（修得後）」：全ドキュメントで統一して使用
- 「習得目安」「習得スキル」「習得した」：README・本文で使用

**問題点：**
- **修得**（しゅうとく）：単位取得・課程修了のニュアンスが強い
- **習得**（しゅうとく）：技能・知識を身につける意味で一般的
- 「学習前後の変化」を示す文脈では**習得前/習得後**の方が自然

**改善提案：**
「あなたがこれを知ると変わること」セクションの「修得前/修得後」を**習得前/習得後**に統一することを推奨。

```
修正例：
- あなた（修得前）→ あなた（習得前）
- あなた（修得後）→ あなた（習得後）
```

## 4. 技術的な誤り・不正確な記述

### 4.1 HTMLドキュメント：`<select>`の誤った説明

**発見箇所：** `docs/03_Web構成要素/01_必須_HTML_Webページの骨組み.md` 206行目付近

```markdown
- **`<select>`**: 選択肢のドロップダウン（都道府県選択など）
...
- 選択肢 → `<select>`（チェックボックス形式）  ← 誤り
```

**問題点：**
`<select>`は**ドロップダウン（プルダウン）**であり、**チェックボックス形式**ではない。チェックボックスは`<input type="checkbox">`。

**改善提案：**
「チェックボックス形式」を削除し、正しい説明に修正。

```
修正例：
- 選択肢 → `<select>`（ドロップダウン形式）、`<input type="checkbox">`（チェックボックス形式）
```

### 4.2 インターネットとWebの世界：ISP説明の補足

**発見箇所：** `docs/02_Web基礎理解/01_必須_インターネットとWebの世界.md` 44行目

**現状：** 「ISP（Internet Service Provider）：インターネットサービスプロバイダー。道路への入口を提供する会社（NTT、ソフトバンクなど）。」

**改善提案：**
document-makerルール「初出で太字＋一言定義」に従い、ISPを初出時に**太字＋簡潔な定義**で明確化。現在は文中説明のみで、太字化されていない箇所あり。用語集（668行目付近）では定義が整理されているため、本文初出時にも同様のスタイルを適用することを推奨。

## 5. ドキュメント作成ルールとの整合性

### 5.1 セクション長（5〜10文目安）

**ルール：** document-maker「1セクションは5〜10文を目安に簡潔に」

**観察：**
- 多くのセクションは適切な長さ
- 一部、応用ドキュメント（例：インターネットとWeb、HTTP通信の基本）で1セクションが15文以上になる箇所がある
- `<details>`で折りたたみされている長い応用説明は許容範囲と判断可能

**改善提案：**
必須・推奨レベルのドキュメントでは、特に「はじめに」直後の導入部分を5〜10文に抑えるよう意識。応用ドキュメントでは折りたたみの活用を維持。

---

### 5.2 Mermaid図の要件

**ルール：** 「シーケンス図とフローチャートを最低1つずつ」

**確認結果：**

| ドキュメント | シーケンス図 | フローチャート |
|-------------|-------------|----------------|
| コンピューターとプログラムの関係 | ✅ | ✅ |
| ファイルとフォルダの世界 | - | ✅ |
| インターネットとWebの世界 | ✅ | ✅ |
| クライアントサーバーモデル | - | ✅ |
| HTTP通信の基本 | ✅ | ✅ |
| HTML Webページの骨組み | - | ✅ |

**改善提案：**
「ファイルとフォルダの世界」「クライアントサーバーモデル」「HTML Webページの骨組み」に、通信の流れや処理の順序を示す**シーケンス図を1つ追加**することを検討。例：クライアントサーバーモデルには、リクエスト→レスポンスのシーケンス図が既存のフローチャートと相性が良い。

---

### 5.3 初出用語の太字＋定義

**ルール：** 「難しい語は初出で太字＋一言定義を付ける」

**観察：**
- 多くの技術用語（タスク、バイナリ、DNS、HTTPメソッドなど）は適切に太字＋説明あり
- 一部、**チューリング完全**（コンピューターとプログラムの関係）などは初出時に一言定義が簡潔で、初学者にはやや難解な可能性
- **NAT**、**CDN**、**RESTful**などは`<details>`内で詳細説明があるものの、初出時の一文定義が弱い箇所あり

**改善提案：**
初出時に「**用語名**：一言で分かる定義」の形式を徹底。例：「**チューリング完全**：理論上、計算できる問題は時間と記憶があれば全て解けるという性質」

## 6. 対象読者層との適合性

### 6.1 技術水準のバランス

**対象読者：** プログラミング未経験のWebディレクター、プランニング/分析業務が主

**良い点：**
- レストラン、郵便、道路などの例えが効果的
- 「あなたがこれを知ると変わること」で実務シーンを具体的に提示
- 開発者との会話例が豊富で、実践的なコミュニケーションをイメージしやすい

**改善余地：**
- 応用ドキュメント（Webセキュリティ、パフォーマンス・SEO）では、「AES-256」「OAuth 2.0」「JWT」など専門用語が初出で一気に登場する箇所がある
- これらは「要件定義の際に使うキーワード」としての文脈は適切だが、一言補足があるとより親切

**改善提案：**
専門用語の羅列が続く箇所では、括弧で「（暗号化方式）」「（認証の標準規格）」などの補足を追加。

---

### 6.2 document-plannerルールとの関係

**ルール：** 「開発者とのやり取りに関しては現状技術的なやり取りは頻繁に発生してはいないので、用いないでください」

**解釈の難しさ：**
- 一方で、「あなたがこれを知ると変わること」では開発者との会話例が多く使われている
- 「用いない」が「例え話を開発者会話に限定しない」の意味なのか、「開発者会話の例自体を減らす」の意味なのかが不明瞭

**現状評価：**
開発者との会話例は、Webディレクターが「知っていると言える状態」を具体的に示す点で学習効果が高い。現状の分量・内容は妥当と判断。

**改善提案：**
document-plannerの該当条項の意図を明確化する文言修正を検討。例：「開発者とのやり取りの例え話は、現場での頻度に合わせて控えめにする」など。

## 7. Windows 11環境の反映

**ルール：** document-reviewer「Windows 11環境の最新の状況に合わせた適切な内容」

**確認結果：**
- ファイルとフォルダの世界：Windows 11の「パスのコピー」機能を明記済み ✅
- ターミナル入門：Windows Terminal の標準搭載、`terminal` 検索での起動を記載 ✅
- 拡張子表示・隠しファイル表示の手順は Windows 10/11 で共通として記載 ✅

**改善提案：**
- 「2025年現在」といった時制表現が複数箇所にある（認証情報、ターミナル章など）。定期的に「最新の状況」として更新する運用を推奨
- Windows 11 の WSL（Windows Subsystem for Linux）への言及は、応用レベルで軽く触れておくと将来の学習拡張に有用

## 9. その他の改善提案

### 9.1 表の可読性

- 長い表（例：インターネットとWebの世界のNAT種類、IPv6形式）は`<details>`で折りたたみ済みで適切
- 一部の表でセル内改行が多く、Markdownレンダリング時に幅が広くなる可能性あり。必要に応じて表を分割することを検討

### 9.2 コードブロックの注釈

**ルール：** 「どの行が何をしているか注釈で説明」

**観察：**
- コンピューターとプログラムの関係のJavaScript例、HTTP通信の基本のPython例など、行単位の注釈が充実している ✅
- 一部の長いコード例では、ブロック全体の「何のためのコードか」の一文説明があるとより分かりやすい

### 9.3 チュートリアル連携

- `tutorials/` 配下に step01〜step05 のチュートリアルと解説ドキュメントが存在
- docs と tutorials の対応関係（例：02_Web基礎理解とstep04_form_postの関連）を、各章の「やってみよう」やREADMEで明示すると、実践との往復がしやすくなる

## 11. 総評

本ドキュメント群は、非エンジニア向けとして十分に配慮された内容であり、例え話の豊富さや「あなたがこれを知ると変わること」による動機づけが特に優れている。技術的厳密さと読みやすさのバランスも概ね良好である。
